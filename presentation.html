The GraphQL Approach

How Bird CLI reads Twitter without the official API

How it works

x.com's frontend fetches all data through an internal GraphQL API. Bird CLI replicates these browser requests using cookie auth — looks like a normal user browsing.

Why it works

  • Twitter can't break this API without breaking their own website
  • Powers x.com since ~2022 — 600M+ monthly users depend on it
  • Query IDs rotate every 2-4 weeks, but Bird CLI auto-detects new IDs from public JS bundles

Twitter's countermeasures

  • Query ID rotation (2-4 wks)
  • Datacenter IP banning
  • TLS fingerprinting
  • Guest token → browser binding
  • Rate limiting (300 req/hr)
  • Cookie validation changes

Risk: Bird CLI may disappear It disappeared. (Feb 8, 2026)

  • Creator took it down — "X now pushes pay-per-use"
  • X reached out to collaborate on a "sustainable path forward" for agents
  • Code lives on in community forks (v0.8.0) — not lost, but unmaintained

Data Collection: What Now?

HTML scraping works for tweets, but not for followers/following — those require the GraphQL layer.

Option Effort Cost Trade-offs
Use fork as-is LOW $0 Works today. No maintainer — breaks when Twitter rotates query IDs (~2-4 wks)
Fork & maintain MED $0 We own the code. Need to track query ID rotation, ~2-4 hrs/month. Must understand Bird CLI internals
Build GraphQL client HIGH $0 Full control, tailored to our needs. Weeks of dev + reverse engineering auth, query IDs, rate limits
Official X API LOW ~$1/acct/mo Stable, legal, supported. Expensive at scale — 500 accounts = $500/mo
Apify actors LOW ~$0.05/acct Managed service, they handle breakage. Platform fee $49/mo + per-result costs
  • Short term: Fork & maintain Bird CLI — we already know the codebase
  • Fallback: Official X API behind provider abstraction — swap in one file
  • Long term: Multi-source (GitHub, LinkedIn) reduces Twitter dependency entirely
